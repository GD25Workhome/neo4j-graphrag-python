# GraphRAG 代码注释与分支速记

> 目标：快速回忆 `src/neo4j_graphrag/generation/graphrag.py` 中 `GraphRAG.search` 与关键分支设计。  
> 使用场景：你已经理解原理，需要 1-2 分钟找回“为什么这么写”。

## 1. 主流程一眼看懂（search）

`search()` 基本就是：

1. 参数校验（含兼容旧参数行为）
2. 处理消息历史（若有）
3. 构造检索 query（可能是“历史摘要 + 当前问题”）
4. 执行 retriever 检索
5. 判断是否走 fallback
6. 若不 fallback：拼 context -> 拼 prompt -> 调 LLM
7. 组装返回结果（可选带检索上下文）

记忆关键词：**检索优先、生成兜底、结果可回传上下文**。

## 2. `_build_query` 的核心意义

- 无历史：直接用 `query_text` 检索。
- 有历史：先把历史对话压缩成摘要，再与当前问题合并，作为检索输入。

快速理解：这是在提升检索 query 的“语义完整度”，避免只用当前一句话导致召回偏差。

## 3. 重点分支：为什么要 `len(retriever_result.items) == 0`

分支逻辑（语义）：

- **检索为空 + 配置了 `response_fallback`** -> 直接返回 fallback 文案，不走 LLM。
- 其他情况 -> 继续正常的 RAG 生成流程。

设计动机速记：

- 避免“空上下文还硬让模型回答”造成幻觉。
- 给调用方可控策略：无证据时要不要“明确拒答”。
- 节省一次无意义的 LLM 调用。

## 4. 什么情况下会出现空检索结果

常见前提：

- 库里确实没有匹配内容（未入库/空库）。
- 相似度与检索阈值条件下无命中。
- metadata/filter 过滤过严导致候选全被筛掉。
- 索引/字段配置不匹配导致可检索集为空。

注意：通常是 `RetrieverResult` 对象存在，但 `items` 列表为空，不是“对象为 None”。

## 5. 你可直接套用的判断框架

当回答质量差时，先看：

1. query 是否被 `_build_query` 正确构造；
2. retriever 是否真的召回到 `items`；
3. 当前场景应不应该配置 `response_fallback`；
4. prompt/context 组装是否过短或噪声过高。

一句话回忆：**这个模块不是“直接问 LLM”，而是“先确认证据，再决定是否生成”。**

# Neo4j 索引与 ANN 理解速记

> 目标：快速回忆“Neo4j 有没有表/索引、向量索引是什么、ANN 为什么能走索引”这条学习链路。

## 1. Neo4j 与关系型的对应关系（先定心智）

- Neo4j 核心数据：`Node` / `Relationship` / `Property`。
- 没有传统“固定列的表”，但有标签（Label）、关系类型（Type）、索引、约束。
- 所以：**不是无 schema**，而是 schema 形态不同。

## 2. `INDEX_NAME` 不是表名

向量索引示例里的 `INDEX_NAME = "chunk-index"`：

- 只是索引对象名称，方便创建、查询、删除、运维。
- 不是“表名”，也不代表 Neo4j 回到了表模型。

## 3. `euclidean` 的准确定位

- `euclidean` 不是索引类型。
- 它是向量索引的相似度/距离度量方式。
- 索引类型是 `VECTOR INDEX`，度量方式是该索引的参数之一。

## 4. 为什么还要“填充向量索引”

文档里的“填充向量索引”更准确是：

- 先建索引定义（label、property、dimension、similarity）；
- 再把 embedding 写入节点属性（upsert）；
- 索引随属性数据进入而变得可检索（可能经历 `POPULATING` 到 `ONLINE`）。

快速记忆：**不是手工往索引结构塞数据，而是写属性值，索引自动纳入。**

## 5. ANN 为什么可以“走索引”

核心不是“完全不算距离”，而是“少算很多对象”：

- 无索引：近似全量向量都要参与距离计算。
- 有 ANN 索引：先用索引结构快速缩小候选，再对少量候选精算/排序。

一句话：**索引不替代计算，索引减少需要计算的候选集。**

## 6. `VectorRetriever` vs `VectorCypherRetriever`

- `VectorRetriever`：标准 ANN TopK 召回主路径（可带 filter）。
- `VectorCypherRetriever`：在 ANN 召回后继续图查询/扩展（`retrieval_query`）。

定位差异：

- 前者偏“向量检索本身”；
- 后者偏“检索后图联动能力”。

## 7. 实战排错最短路径

当效果不对时，按顺序查：

1. 维度是否匹配（embedding 长度 vs index dimensions）；
2. 属性名/标签名是否一致；
3. 相似度函数与预期是否一致；
4. 过滤条件是否过严；
5. 索引状态是否在线、数据是否已写入。
